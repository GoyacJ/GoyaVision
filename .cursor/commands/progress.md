# progress

更新开发进度文档的指南。完成功能或重要里程碑后使用。

## 更新文件

`docs/development-progress.md`

## 状态标记

- ✅ **已完成** - 功能已实现、测试通过、文档更新
- 🚧 **进行中** - 正在开发，部分功能可用
- ⏸️ **待开始** - 计划中但尚未开始
- ⚠️ **阻塞** - 遇到阻塞问题，需要解决
- 🔄 **重构中** - 功能可用但正在重构优化

## 更新步骤

### 1. 找到对应的功能模块

开发进度文档通常按功能模块组织：
- 核心功能（媒体源、资产、算子、工作流、任务、产物）
- 认证与权限（JWT、RBAC）
- 前端页面
- 部署与运维
- 测试覆盖

### 2. 更新状态

将相关功能的状态标记从一个阶段更新到另一个阶段。

**示例**:
```markdown
## 核心功能

| 功能 | 状态 | 说明 |
|------|------|------|
| 媒体资产管理 | ✅ | 已完成 CRUD、搜索、派生追踪功能 |
| 算子管理 | 🚧 | CRUD 已完成，版本管理开发中 |
| 简化工作流 | 🚧 | Phase 1：单算子任务进行中 |
| 任务调度 | ⏸️ | 待开始 |
```

### 3. 添加说明信息

在"说明"列中提供详细信息：
- **已完成的子功能** - 列出已实现的具体能力
- **当前进度** - 进行中的功能完成了哪些部分
- **技术债务** - 已知问题或待优化项
- **阻塞原因** - 如果阻塞，说明原因

**示例**:
```markdown
| 功能 | 状态 | 说明 |
|------|------|------|
| 媒体资产管理 | ✅ | 已完成 CRUD、标签系统、派生追踪、搜索过滤功能 |
| 算子管理 | 🚧 | CRUD 已完成，版本管理开发中。待实现：自定义算子上传 |
| 工作流设计器 | ⏸️ | 待开始。依赖：前端 DAG 可视化库选型 |
| 复杂工作流 | ⚠️ | 阻塞：DAG 并行执行引擎设计评审中 |
```

### 4. 更新完成百分比（如适用）

如果文档中跟踪整体进度百分比，更新相应数值。

**示例**:
```markdown
## V1.0 整体进度

- 已完成: 45%
- 进行中: 35%
- 待开始: 20%
```

### 5. 添加时间戳（可选）

在重要里程碑处添加完成时间。

**示例**:
```markdown
| 功能 | 状态 | 说明 | 完成时间 |
|------|------|------|----------|
| 媒体资产管理 | ✅ | 已完成 CRUD、搜索、派生追踪功能 | 2024-01-15 |
```

## 更新时机

### 必须更新的情况

- ✅ **完成主要功能** - 功能从 🚧 进行中 → ✅ 已完成
- 🚧 **开始新功能** - 功能从 ⏸️ 待开始 → 🚧 进行中
- ⚠️ **遇到阻塞** - 功能被阻塞，需要标记并说明原因

### 可选更新的情况

- 完成子功能时更新"说明"列
- 解决技术债务时更新备注
- 调整优先级或计划变更时

## 配合其他命令

开发进度更新通常与以下命令一起使用：

1. `/dev-done` - 完成开发后自动提醒更新进度
2. `/commit` - 提交时在 commit message 中引用进度变更
3. `CHANGELOG.md` - 进度更新后同步到变更日志

## 注意事项

- 状态标记要准确反映真实情况
- 说明要简洁但足够详细
- 重要功能完成时添加完成时间
- 遇到阻塞立即标记并说明原因
- 定期回顾和更新整体进度
